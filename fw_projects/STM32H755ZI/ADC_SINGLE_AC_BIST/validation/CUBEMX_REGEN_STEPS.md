# Peripheral Integration Steps

Use these steps only if the user chooses CubeMX regeneration after focused ADC/DAC/timer/DMA resource discovery. The existing CM7 and CM4 `Core/Src/main.c` files must be used; do not create another `main.c`.

## Why Peripheral Integration Is Needed

The current generated project confirms PA4 as `DAC1_OUT1` and `ADC1_INP18`, and `.mxproject` references the existing CM7/CM4 `main.c` files. However, ADC1 and DAC1 are not enabled as configured peripherals. The generated driver set currently includes TIM and DMA HAL sources, but not ADC and DAC HAL sources. This absence means the agent must ask focused resource questions first; editing only `main.c` or `stm32h7xx_hal_conf.h` would leave the project incomplete and fragile.

## Project To Open

Open this IOC in STM32CubeMX or STM32CubeIDE:

```text
fw_projects/STM32H755ZI/ADC_SINGLE_AC_BIST/ADC_SINGLE_AC_BIST/ADC_SINGLE_AC_BIST.ioc
```

## Core Ownership

Select the core that owns the BIST resources before generating code.

Recommended default for this trial: CM7 owns ADC1, DAC1, timer, and DMA resources.

If the safety architecture requires CM4 ownership, select CM4 consistently for all BIST peripherals instead.

## Required Peripheral Configuration

1. Enable ADC1.
   - Input: PA4 / ADC1_INP18.
   - Conversion group: regular conversion.
   - Capture count target: 256 samples in BIST code.
   - External trigger: select a timer trigger that CubeMX supports for ADC1 on STM32H755.

2. Enable DAC1.
   - Channel: DAC1_OUT1 on PA4.
   - Trigger: timer trigger from the same timing base as ADC acquisition.
   - DMA: enable DMA transfer for DAC1 channel 1.

3. Enable a timer time base.
   - Use a deterministic trigger source supported by both the DAC and ADC configuration.
   - Prefer a simple timer trigger generated by CubeMX rather than manual register setup.
   - The ADC trigger must be deterministic relative to the DAC update; if no delayed trigger is configured yet, document it as an open timing item.

4. Enable ADC DMA.
   - Request: ADC1 DMA request as generated by CubeMX/DMAMUX.
   - Direction: peripheral to memory.
   - Mode: normal for exactly one 256-sample capture unless CubeMX constraints require another generated mode.

5. Enable DAC DMA.
   - Request: DAC1 channel 1 DMA request as generated by CubeMX/DMAMUX.
   - Direction: memory to peripheral.
   - Mode: normal or circular according to the final acquisition start/stop design; for the first deterministic BIST trial, normal is preferred if CubeMX supports the desired waveform length.

## Regenerate Code

Regenerate code only after the ADC/DAC/timer/DMA choices are confirmed and the user selects the CubeMX path. If the user selects manual integration instead, document the equivalent integration evidence before editing firmware. Keep user code enabled if CubeMX is used.

## Evidence Expected After Regeneration

After regeneration, verify that the selected owner core contains:

- `MX_ADC1_Init`
- `MX_DAC1_Init`
- `MX_TIMx_Init` for the selected trigger timer
- DMA initialization for ADC and DAC
- ADC, DAC, TIM, and DMA HAL modules enabled in `stm32h7xx_hal_conf.h`
- ADC and DAC HAL driver source/header files included or linked by the generated project

## BIST Code Still Not Added Yet

Only after this evidence exists should BIST firmware code be added. The BIST code must remain bounded, static-memory only, diagnosable, and RAM-executed according to the project rules.
